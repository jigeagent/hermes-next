# Hermes Next v0.4 — Feedback 闭环 + 启动恢复

> 对标：memos-local v2.0 的 `submitFeedback` → `runFeedbackExperience` → Decision Repair
> 目标：让 hermes-next 从"单向记录"进化为"双向学习"

---

## 一、Feedback 闭环

### 1.1 现状（v0.3.x）

```
用户 ↔ agent 对话 → sync_turn → capture_trace → SQLite + OV
                                ↓
                         on_session_end → Reward → L2 → L3 → Skill
```

**问题**：反馈只发生在会话结束时，用户给个 👍/👎 不会立即影响检索质量。L2 Policy 一旦归纳出来就冻住了，不会因为后续的负反馈而调整。

### 1.2 目标（v0.4）

```
用户 ↔ agent 对话 → sync_turn → capture_trace
       ↓                              ↓
  用户反馈 👍/👎              on_session_end
       ↓                        Reward → L2 → L3
  submit_feedback()
       ↓
  Reward 重算           ← 立即影响权重
  L2 重归纳             ← 模式调整
  Decision Repair       ← 修复写入 Policy @repair 块
       ↓
  prefetch() 检索时     ← Policy 带 repair 指南注入上下文
```

#### 核心流程

def submit_feedback(self, signal: FeedbackSignal) -> None:
    """接收用户反馈 -> 更新 Reward -> 触发 L2 重归纳 -> Decision Repair

    防抖：同一 session 30 秒内合并反馈，只触发 1 次 L2 重归纳
    门槛：magnitude < 0.3 的弱反馈只写 @repair，不触发重算
    多 Agent：每个 Agent 的 feedback 只影响自己的 Policy，不交叉
    """
    # 1. 持久化反馈
    self._feedback_repo.insert(signal)

    # 2. 弱反馈（magnitude < 0.3）只写 repair 不动 reward
    if signal.magnitude < 0.3:
        if signal.polarity == "negative" and signal.text:
            self._decision_repair(signal)
        return

    # 3. 防抖：同一 episode 30 秒内的同类反馈合并
    recent = self._feedback_repo.count_recent(
        episode_id=signal.episode_id,
        within_seconds=30,
        same_polarity=signal.polarity,
    )
    if recent > 0:
        return

    # 4. 奖励重算
    outcome = OutcomeSignal.USER_THUMBS_UP if signal.polarity == "positive" \n        else OutcomeSignal.USER_THUMBS_DOWN
    updated_traces = self._pipeline.reward_engine.apply_outcome(
        traces=self._session_traces,
        signal=outcome,
        target_turn_index=signal.trace_id,
        manual_value=signal.magnitude,
    )

    # 5. L2 重归纳（累积 >= 3 次负反馈才触发）
    if signal.polarity != "neutral":
        recent_negatives = self._feedback_repo.count_recent(
            signal.episode_id, polarity="negative", hours=24,
        )
        if recent_negatives >= 3:
            new_policies = self._pipeline.policy_inducer.batch_induce(
                traces=updated_traces,
                existing_policies=self._pipeline.policies,
            )
            for p in new_policies:
                self._policy_repo.insert(p)
                self._native.promote_policy(
                    p.name, p.trigger_pattern, p.action_template, p.confidence,
                )

    # 6. Decision Repair (负反馈 -> Policy @repair)
    if signal.polarity == "negative" and signal.text:
        self._decision_repair(signal)

```python
def _decision_repair(self, signal: FeedbackSignal) -> None:
    """用户纠正 → 提取偏好/反模式 → 写入相关 Policy 的 repair 块

    分两阶段：
    v0.4-alpha：按 episode_id → trace → 时间窗口匹配相关 Policy
    v0.4-beta：embedding 语义匹配 + scope 字段
    """
    if not signal.text:
        return

    policies = self._policy_repo.list_active()
    if not policies:
        return

    # v0.4-alpha：关键词 + 时间窗口匹配
    matched = []
    for p in policies:
        # 检查 signal.text 中的关键词是否出现在 policy 的 trigger 或 description 中
        keywords = set(signal.text.lower().split())
        trigger_words = set(p.trigger_pattern.lower().split())
        if keywords & trigger_words:
            matched.append(p)

    # 取匹配度最高的 policy（关键词交集最大）
    if not matched:
        return
    target = max(matched, key=lambda p: len(set(signal.text.lower().split()) & set(p.trigger_pattern.lower().split())))

    # 写入 repair 块（含 scope 字段）
    # 格式:
    #   @repair
    #     preference: [用户偏好的做法]
    #     anti_pattern: [用户反对的做法]
    #     scope: global | scene-specific
    #     source: feedback_{id}
    repair_entry = {
        "preference": [],
        "anti_pattern": [signal.text],
        "scope": "global",  # 默认全局，后续可由 LLM 判断
        "source": f"feedback_{signal.episode_id}",
    }
    # 追加到现有 repair 块
    existing = target.metadata.get("repair", [])
    existing.append(repair_entry)
    self._policy_repo.update_metadata(target.id, {"repair": existing})
    logger.info("Decision Repair applied to policy %s: %s", target.name, signal.text[:60])
```

> 灵儿建议加 `scope` 字段：
> - `scope: global` → 所有场景生效（如"禁止渐变背景"）
> - `scope: scene-specific` → 仅当前项目/场景生效（如"大国中医不用冷色调"）
> v0.4-alpha 默认 global，v0.4-beta 加场景推断。

## 二、启动恢复

### 2.1 问题

当前进程重启后：
- 未完成的 session 丢失
- 正在进行的认知管道中断
- 已经 sync_turn 但 on_session_end 未触发的 trace 缓存丢失

### 2.2 方案

#### 新增 `session_state` SQLite 表

```sql
CREATE TABLE IF NOT EXISTS session_state (
    session_id    TEXT PRIMARY KEY,
    agent_name    TEXT NOT NULL,
    turn_index    INTEGER NOT NULL DEFAULT 0,
    status        TEXT NOT NULL DEFAULT 'open',  -- open | closed
    opened_at     TEXT NOT NULL,
    last_active   TEXT NOT NULL,
    metadata      TEXT DEFAULT '{}'
);
```

#### initialize() 时恢复

```python
def initialize(self, session_id: str, **kwargs):
    # ... 现有初始化逻辑 ...

    # 启动恢复：扫描未完成的 session
    if self._cache:
        self._recover_orphan_sessions()

def _recover_orphan_sessions(self):
    """扫描 stale session → 执行 session_end 流程 → 触发认知管道

    stale 阈值可配置：
    - 默认 4h（适合日常对话场景）
    - 好妹（Linux VM，可能跑几天不关机）可调高到 24h
    - 灵儿（频繁重启）可调低到 1h
    """
    stale_hours = self._config.lifecycle.session_stale_hours
    stale = self._session_repo.list_open_stale(hours=stale_hours)
    for session in stale:
        logger.info("Recovering orphan session: %s", session.session_id)
        # 1. 加载该 session 的 traces
        traces = self._trace_repo.list_by_session(session.session_id)
        if not traces:
            self._session_repo.close(session.session_id)
            continue

        # 2. 执行 session_end 流程（Reward → L2 → L3）
        results = self._pipeline.process_session_end(
            traces=traces,
            session_success=True,
        )
        self._persist_pipeline_results(results)

        # 3. 标记 session 已关闭
        self._session_repo.close(session.session_id)
        logger.info("Recovered session %s: %d traces, %d policies",
                     session.session_id, len(traces),
                     len(results.get("new_policies", [])))

def sync_turn(self, ..., session_id):
    # ... 现有逻辑 ...
    # 新增：更新时间戳
    if self._session_repo:
        self._session_repo.touch(session_id)
```

def initialize(self, ...):

### 2.3 会话恢复流程图

```
进程启动 → initialize()
            ↓
    检测 session_state 表
     ├── open 且 last_active > 4h → 视为 stale
     │     └── 加载 traces → process_session_end() → 关闭
     └── open 且 last_active ≤ 4h → 视为活动
           └── 保留，等待下一轮 sync_turn 续接

正常流程 → sync_turn()
            ↓
    更新 session_state.last_active = now

正常关闭 → on_session_end()
            ↓
    标记 session_state.status = "closed"
```

## 四、总结

两个功能补上后，hermes-next 的叙事链就完整了：

```
v0.1-0.2:  基础存储 + 检索               "能记住"
v0.3:      认知管道 + 晋升 + 原型记忆融合  "会归纳"
v0.4:      Feedback 闭环 + 启动恢复       "会学习"
v0.5:      Decision Repair + Hub 共享    "会修复"
```

Feedback 闭环是**从单向记录到双向学习**的质变——用户给个 👍/👎，系统下次就知道该怎么做。这是 hermes-next 和 memos-local 拉开的真正差距。
